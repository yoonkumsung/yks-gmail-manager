## 0. 한 줄 결론

품질 문제의 **가장 깊은 근본원인은 "LLM이 풀 문제"와 "룰베이스(코드/JSON)로 결정해야 할 문제"의 경계가 무너진 것**이다.
출처·라벨라우팅·중복제거·정크필터·렌더모드는 **결정적(deterministic)** 이어야 하는데 지금은 LLM 자유생성에 맡겨져 있고, 그 결과 같은 뉴스레터가 실행마다 다르게 나온다. 반대로 LLM이 잘해야 할 요약합성/번역/의미병합은 프롬프트가 약하거나(혹은 파일이 파손돼) 제대로 안 된다.

## 2. 정량 데이터 증거 (6/1 실제 추출본)

`output/backfill/20260501/` 의 병합 전(`labels/*/items`) vs 병합 후(`merged/`) 비교:

```
PRE-MERGE  : 557 items | source 누락 0건 | (Haiku 추출은 source 100% 채움)
POST-MERGE : 555 items | 병합으로 제거된 중복 단 2건 (경제)
```

- **출처는 추출 단계에선 다 있다** → 누락은 "추출"이 아니라 **(a) 프로덕션 Ollama 추출의 불안정성, (b) 파손된 라벨 에이전트, (c) LLM 철자 드리프트**에서 발생.
- **병합(dedup)이 사실상 작동하지 않음** — 555/557, 제거율 0.4%.
- **추출 아이템 키 = `source,summary,title,link,keywords`** — `source_email`/`message_id`가 backfill 경로엔 없음(orchestrator 본선만 enrich). 결정적 키가 경로마다 다름.
- **link 트래킹 리다이렉트**: `event.stibee.com/v2/click/...` 동일 도메인이 서로 다른 기사 → 단순 prefix 매칭 dedup은 위험, **리다이렉트 해석 필요**.
- **영문 제목 잔존 43건**(스타트업12, 경제11, IT6, 소셜포럼5 …) — 번역 누락.
- **혼종 CJK**: "평범함을 문화로 変える" 등 일본어 한자 누출.

## 3. 근본원인 목록 (RC-A ~ RC-J)

### RC-A — 출처가 LLM 자유생성이라 불안정 〔RULE로 이관〕
- **증상**: 출처 누락/들쭉날쭉(문제 2,3,8,14), 같은 뉴스레터가 다른 이름으로 분리(SCMP 2개, 에이스퀘어 4철자).
- **원인**: 라벨 에이전트가 `"source":"뉴스레터 이름"`을 LLM에 출력시킴. LLM은 (a)비우거나 (b)기사 발행처(etnews.com)를 넣거나 (c)한글 철자를 깨뜨림. orchestrator는 `source_email`만 enrich하고 **신뢰가능한 source 이름을 세팅 안 함**. render는 `item.source||''` 그대로.
- **해결 계층**: **RULE** — orchestrator enrich에서 `newsletters.json`의 name을 `source_email`로 조회해 모든 아이템에 `source`를 결정적으로 주입. render는 없으면 source_email fallback.

### RC-B — link 기반 중복제거 부재 〔RULE 신규〕
- **증상**: 티저+원문 중복(문제 12,13), 한 기사 분리(1,9), 동일내용 2번(6), 트렌드라이트 같은 사건 중복(3).
- **원인 1**: `fetch_articles.js`가 원문을 `=== 원문 기사 전문 ===` 별도 섹션으로 append → LLM이 티저와 원문을 각각 아이템화. 제목이 달라 title유사도(0.75) dedup을 빠져나감.
- **원인 2**: 병합 에이전트 규칙 "같은 source_email끼리 분리 유지"(`병합.md`)가 **같은 뉴스레터 내 티저↔원문 병합을 원천 차단**.
- **원인 3**: `link` 동일성 기반 dedup이 코드 어디에도 없음. dedup 전부 title기반.
- **해결 계층**: **RULE** — 병합 직전 link 정규화(utm/쿼리 제거, 트래킹 리다이렉트 해석) 후 동일 link 병합(더 긴 summary 채택). 같은출처-분리 규칙은 link/제목 동일 시 예외 허용.

### RC-C — 라벨 간 중복: 가드가 침묵 실패 〔RULE 강화〕 ★실측 검증됨
- **증상**: Morning Brew가 경제+미국_경제 양쪽(문제 4), 반려동물 중복(1).
- **원인(실측)**: 병합은 **라벨 내부에서만** 수행. 라벨간 방어는 `filterRawByCatalogLabel` 가드가 유일. **6/1 raw 데이터를 직접 검사한 결과 Morning Brew가 경제·미국_경제 양쪽 raw에 존재하며 둘 다 `gmail_labels: undefined`.** 가드는 `gmail_labels`가 없으면 `if (!gmailLabels) continue`로 **경고 없이 no-op** → 양쪽 유지. 즉 가드가 작동조차 안 함. (해당 backfill은 gmail_labels 저장 도입 이전 생성본.)
- **핵심 취약점**: gmail_labels가 비는 모든 상황(구버전 데이터, 라벨 API 실패, getLabelIdNameMap 빈 맵)에서 라벨 dedup이 **침묵 실패**하고 텔레메트리도 없음.
- **해결 계층**: **RULE** — (1) 가드가 gmail_labels 부재로 스킵할 때 **경고 로그**, (2) **통합 리포트 직전 전체 라벨 교차 link/제목 dedup 안전망**(가드와 무관하게 동작), (3) 신규 run은 gmail_labels 채워지는지 검증.

### RC-D — 무의미한 한줄 아이템 〔RULE + LLM〕
- **증상**: "밝힌 이유", "이슈한입:앤트로픽", "Mays and Emma Goldberg"(문제 8,11,15).
- **원인**: 추출이 섹션헤더/기자바이라인/목차 조각을 아이템화. `mergeChunkResults`는 `title.length<3`만 필터. summary 공란·제목≈요약·링크없음 조합 필터 없음.
- **해결 계층**: **RULE**(보수적 코드 가드: summary 없음 or ===title 이며 동사없는 짧은 명사구 제거) + **LLM**(에이전트에 "헤더/바이라인/목차는 아이템 아님" 명시).

### RC-E — 번역 누락 〔LLM 규칙 강화 + 파손복구〕
- **증상**: 소셜포럼/Reddit 영문 그대로(문제 5), 영문 제목 43건.
- **원인**: (1) **라벨 에이전트 8개 파손**으로 번역규칙·출력스키마 소실(아래 RC-I). (2) 해외 뉴스레터 SKILL 18개 중 **10개가 언어=ko 오표기 또는 번역규칙 없음**(redditmail, axios, axios_mike, scmp_victoria_bela, hypebeast, joinsuperhuman, dailybyte, wendy_wu …). (3) 전역 systemPrompt 번역 규칙 약함.
- **해결 계층**: **LLM** — 파손 에이전트 복구 + SKILL 언어플래그/번역규칙 일괄 수정 + systemPrompt 번역 강제 강화.

### RC-F — 렌더모드(목록/카드/브리핑) 미분류 〔RULE/CONFIG〕
- **증상**: 스타트업 긴기사+짧은투자소식 뒤섞임(문제 7), 지원사업/센서블박스를 항목마다 파싱(문제 10).
- **원인**: 렌더 분류가 message_id 그룹 휴리스틱(`render_report.js` `g.length>=25 && avg<110`)뿐. 뉴스레터 **유형 메타데이터가 없음**. 지원사업은 `labels.json`에 enabled=true(※ CLAUDE.md엔 disabled — drift). 센서블박스는 PDF라 추출불가.
- **해결 계층**: **RULE/CONFIG** — 카탈로그/SKILL에 `render_mode`(card|list|briefing) 메타 추가. 감사 결과: KDI·KIF=list, SBA·지원사업=briefing, 나머지=card. 스타트업은 아이템단위 "짧은 투자유치형→접이식 목록" 분류.

### RC-G — 리포트 인덱스 라우팅 404 〔RULE〕
- **증상**: `/reports/`에서 날짜 클릭 → `/reports/reports/2026-05-30.html` 404(문제 16).
- **원인**: `generate_index_page.js`가 **동일 HTML을 root와 reports/ 양쪽에 기록**(222-223행). 링크는 항상 `href="reports/FILE"`(84행). reports/ 안에서 누르면 경로 중복.
- **해결 계층**: **RULE** — reports/index.html은 prefix 없는 `href="FILE"`로 별도 생성.

### RC-H — 코드 경로별 enrich 불일치 〔RULE〕
- **증상**: 잠재적. backfill 아이템에 `source_email`/`message_id` 없음 → Gmail링크·교차dedup 불가.
- **원인**: enrich가 `orchestrator.processLabel`에만 있고 backfill/QA 경로엔 없음.
- **해결 계층**: **RULE** — enrich를 공용 함수로 추출해 모든 경로가 동일 메타데이터 보장.

### RC-I — 라벨 에이전트 8개 파일 파손 〔복구〕
- **증상**: 출처 누락이 시사·마케팅·라이프 등에 집중. 번역 누락.
- **원인(검증됨)**: `경제, 기타, 라이프, 마케팅, 소셜포럼, 스포츠, 시사, 인문학` 8개가 `## 입력`/`## 출력` 섹션에 **백슬래시 단독줄 2개**만 남고 **출력 JSON 스키마(source 포함) 소실**. 나머지 14개(NYT/미국/중국/글로벌/IT/스타트업/지원사업)는 정상. 과거 일괄편집(sed/정규식)이 코드블록을 `\`로 치환한 것으로 추정.
- **해결 계층**: **복구** — 8개 파일의 입력/출력 스키마 복원. 차제에 _공통규칙.md 단일소스 참조 구조로 DRY화(현재 22개 파일에 규칙 중복).

### RC-J — 품질 LLM 잡음 (혼종 CJK 등) 〔LLM〕
- **증상**: "変える" 등 일본어/중국어 한자 누출, 이모지 제목("This is just sad 🥲").
- **원인**: 모델 출력 잡음. 후처리 정규화 없음.
- **해결 계층**: **RULE**(후처리: 한국어 본문 내 かな/특정 한자 감지·교정 또는 재시도 플래그) 보조 + **LLM** 프롬프트.

### RC-K — 카탈로그 메타데이터 stale/중복 〔CONFIG 정리〕 ★전수 census로 발견
- **증상**: render_mode 자동분류 불가, 출처 중복.
- **원인(실측)**: `newsletters.json`의 `structure.item_count_avg`가 실제와 크게 어긋남 — KDI=3(실제135, 45배), SCMP=20(실제60), StrictlyVC=15(실제39), 서울시청=1(실제19). Morning Brew는 **중복 엔트리**(avg 20/8/3)로 존재. 즉 카탈로그 볼륨 메타를 신뢰해 분류하면 오분류.
- **해결 계층**: **CONFIG** — render_mode를 item_count_avg가 아니라 **명시 큐레이션 필드 또는 런타임 볼륨**으로 결정. 중복 카탈로그 엔트리 정리. (단 structure.type[multi-item/single-topic]은 비교적 신뢰 가능 → card vs list 1차 분류엔 사용 가능.)

### RC-L — 정적 검증(validate_skills)이 얕음 〔검증 강화〕
- **증상**: 파손 8개·번역결핍 22개·stale 메타를 **자동검증이 전혀 못 잡음**(`validation_report.json`: issues:[], warnings:[]).
- **원인**: `validate_skills.js`는 발신자↔카탈로그 일치만 점검. 파일 파손/언어플래그/번역규칙/스키마 무결성 미점검.
- **해결 계층**: **검증** — validate_skills에 (a)라벨 에이전트/SKILL 파손(백슬래시 단독줄·빈 스키마) 검사, (b)해외라벨인데 번역규칙 없음 경고, (c)structure 무결성 추가.

### RC-M — 정보 누락(1원칙 위배) 〔파이프라인 다지점〕 ★콘텐츠 대조로 발견
- **증상**: 중국_시사 Daily Briefing 29→11(~14건 소실), 마케팅 큐레터 단신 5건, 라이프 Hypebeast 5건, 시사 동아 1건 누락.
- **원인(복합)**: (1) **크롤러가 대형 브리핑 메일의 상위 기사만 따라가고 나머지 헤드라인 폐기**(maxUrls=15 상한 + 본문 헤드라인-only 항목 미보존). (2) SKILL이 `◾`/Briefing 단신 섹션을 추출규칙에 안 넣거나, 넣어도 LLM이 누락. (3) 청킹 경계 손실 가능.
- **해결 계층**: **RULE+LLM** — 본문 헤드라인 항목은 크롤 성공/실패와 무관히 **전부 아이템으로 보존**(링크만 있어도). SKILL에 단신섹션 명시. 추출 후 "본문 헤드라인 수 vs 추출 수" 카운트 검증·경고.

### RC-N — 크롤링 오염(footer/지난호/홍보 무차별 수집) 〔RULE〕 ★발견
- **증상**: 소셜포럼 CEO메일이 외부 신문 5건 정크화, 시사 SELUB가 과거 4/20~4/27 콘텐츠 9건 주입(날짜 오염), IT 지피터스 footer(채용·오픈채팅) 3건.
- **원인**: `enrichWithArticles`가 본문 기사 링크와 **footer/네비/"지난 레터 보기"/홍보 링크를 구분 없이 크롤링**. filterArticleUrls 제외목록이 얕음. 크롤 결과가 별도 섹션으로 붙어 LLM이 뉴스로 추출.
- **해결 계층**: **RULE** — (1) 크롤 대상을 "본문 아이템에 연결된 링크"로 제한(앵커 텍스트/위치 기반), footer·지난호·구독·홍보 링크 강력 배제. (2) 크롤 도메인이 발신 뉴스레터와 무관하면 가중치↓. (3) 날짜 오염 방지: 지난호 링크 패턴(`지난 레터`, `지난호`, `View past`) 차단.

### RC-O — 할루시네이션(인명/수치/날조) 〔LLM 강화 + RULE 검출〕 ★발견
- **증상**: 김범석→"김봉진", Apple $57B→"57억"(10배), 브렌트유 "OPEC 차질" 날조, Harvey $11B→"114억", 본문없는 티저에 요약 날조.
- **원인**: (1) 단위 환산($B→억/조) 오류 빈발. (2) 본문이 거의 없는 항목(크롤 실패/티저)에 LLM이 배경지식으로 채움. (3) 고유명사 혼동.
- **해결 계층**: **LLM**(프롬프트에 단위환산 규칙·"본문 없으면 요약 생성 금지" 강화) + **RULE**(달러·원 수치 환산 사후 검증, 본문 길이 대비 요약 과다 시 플래그).

### RC-P — html_to_text 본문 전소실 〔파서 복구〕 ★발견
- **증상**: 기타 녹색소비자연대 2메일 clean_text 9자/0자, 지원사업 경기RISE 빈 표 셀만 → 발신처 전건 증발.
- **원인**: 특정 HTML 구조(테이블 레이아웃 등)에서 `html_to_text`가 본문을 통째로 버림.
- **해결 계층**: **RULE** — 해당 메일 패턴 진단·파서 보강. clean_text가 비정상적으로 짧으면(예: <50자인데 html_body는 큼) **경고 + subject 폴백 + 원문 재파싱 시도**.

### RC-Q — 라벨 fetch가 silent하게 0건 (고볼륨 라벨 통째 증발) 〔RULE+알림〕 ★실측
- **증상(실측)**: 6/1 백필이 NYT_시사를 `total_fetched: 0`으로 기록. 그러나 직접 Gmail 조회 시 **NYT/시사는 최근 14일 246건, 6/1 윈도우에도 10건 존재**(라벨·쿼리 정상, 재수집으로 10건 fetch 성공). 즉 백필 실행 당시 fetch가 0을 반환하고 **에러 없이 "메일 없음"으로 기록**됨.
- **원인**: `fetchGmailMessages`는 비인증 오류를 `console.warn` 후 흡수(null/빈배열). 일시적 API 오류·빈 응답이 **고볼륨 라벨의 전건 누락으로 이어지나 경고만 찍히고 파이프라인은 green**. 과거 silent-green 인시던트와 동족.
- **영향**: NYT(하루 다수 에디션·고볼륨)가 통째로 빠져도 리포트는 정상처럼 생성됨. 정보 누락 제로 원칙 정면 위배.
- **해결 계층**: **RULE+알림** — 라벨별 기대 볼륨 대비 0건/급감 시 경고·실패 처리. 카탈로그에 라벨이 등록 발신자를 가지면 0건일 때 의심. fetch 결과를 progress에 남겨 사후 감사 가능하게.

### RC-R — NYT류 멀티에디션 라벨 내 대량 중복 〔RULE dedup 강화〕 ★실측
- **증상(실측)**: NYT는 하루 10개 에디션(The Morning/Evening/For You/Today's Headlines/Breaking 등)을 보내 **같은 기사가 에디션 간 반복**(재추출 결과 Tilly Norwood 5중복, IRS·이란 등). 라벨 내 dedup이 없으면 다이제스트가 동일 기사로 도배.
- **원인**: 병합이 라벨 내부에서 돌지만 link/제목 기반 dedup이 약해 cross-에디션 중복을 못 잡음(RC-B와 동근). For You 헤드라인-스킵 규칙과 누락제로가 충돌(다른 에디션엔 요약 존재).
- **해결 계층**: **RULE** — link 기반 dedup(RC-B)이 cross-에디션 중복도 흡수. 헤드라인-only는 같은 link의 요약본과 병합.

> **RC-D 정밀화(바이라인 파서)**: NYT가 "헤드라인+부제+`By 저자명`"을 **하나의 마크다운 링크 텍스트로 묶어** 보냄. 줄/콤마 split 파서가 `By` 이후 저자명을 별도 한줄 아이템("Mays and Emma Goldberg")으로 만든다. 규칙대로 "링크텍스트=1아이템, `By` 이후=summary 흡수"하면 미발생(재추출 시 정크 0 확인). The Morning/Magazine/World/Wirecutter 전반에 동일 위험.
