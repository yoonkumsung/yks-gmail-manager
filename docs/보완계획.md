## B-P. 성능 — 실행 속도 (2026-06-25 측정 기반, 런 무관 수정)

**측정 (deepseek-pro 전수 런 로그, 편향 없는 지표):**
- 평균 동시성 = busy/wall = 762×8.46s / 9,133s ≈ **0.71** — `concurrencyLimit:3`인데 실효 0.71(거의 직렬 + ~29% 유휴).
- 호출시작 간격 중앙값 6.68초 ≈ 호출 1건 소요(8.5s) → 이전 호출 끝나야 다음 시작 = 직렬.
- 호출 1건 8.5s 중 **7.9s가 deepseek 생성**(첫바이트 1.4s). rate-gate(2s)·rate-limit는 **무관**(실제 간격 6.68s ≫ 2s, 대기 152분 중 2.8분뿐).

**근본 원인:** ① deepseek-pro 생성 8.5s/건 ② 3-라벨 동시성이 LLM 병렬로 안 이어짐 — 라벨 내부 추출이 순차(`for(cleanFiles)` + 메일 내 청크 순차)이고, 무거운 비-LLM 크롤(155회)이 슬롯을 점유, 3슬롯이 서로 다른 단계(크롤/추출/병합)에 흩어져 어느 순간에도 LLM 도는 라벨 ~1개.

| ID | 작업 | 계층 | 기대효과 |

| BP-1 | **라벨 내 청크/메일 추출 병렬화**: 현재 순차 `for` → 동시 N개(예: 청크 단위 `Promise.all` + 동시 상한). 토큰/출력 안전 유지 | RULE | LLM 풀가동, ~3배 단축(라벨 1개로도 병렬) |

### C-해결책 논의 (미확정 — 기록용)
텍스트 무결성(C-O/C-P) 해결책으로 3안 토론:
- **표적 가드**(무원문 가드 + 전체 대조 검증): 저비용이나 ⓑ 고볼륨 꼬리 못 잡음 + 전체 대조 false-negative → **불충분 판정.**
- **앵커-포인터 개편**(LLM은 가리키고 코드가 복사): 구성적 보존이나 추출 계약 전면 개편 + 앵커매칭 실패 시 누락위험 → **과함 판정.**
- **사후 아이템별 localization 검증**(코드가 사후에 각 아이템 원문 span 탐색, 닻 없으면 플래그): 계약 불변·누락위험 없음·두 메커니즘 다 포착 → **유력하나 미확정.**
→ 다음 세션에서 이 중 확정 후 C-O/C-P 대책 정식화.

### C-병합 임계값 (확정)
유사도 분포상 실제 병합이 0.3~0.6 집중, 고유사도(0.8~0.9) 병합 5건뿐. → **현 임계값 0.25 유지**(상향 시 진짜 병합 놓침), **near-exact 자동병합 밴드 불필요**. 문제는 임계값이 아니라 **link/메타 무결성**(C-LINK)이 dedup을 막는 것 → C-B로 해결.
