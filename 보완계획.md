> 원칙: **결정적으로 정해지는 값은 룰베이스(코드/JSON), 글의 의미가 필요한 일만 LLM.**

## 1. 설계 원칙 — LLM vs 룰베이스 경계

| 책임 | 담당 | 이유 |
|---|---|---|
| 출처(뉴스레터명) | **RULE**: 카탈로그 `newsletters.json`에서 발신자→name | 발신자만 알면 100% 결정. LLM은 철자 드리프트(에이스퀘어 4종)·발행처 혼동 발생 |
| source_email / message_id / received_at | **RULE**: enrich 공용함수 | 메타데이터는 추출과 무관 |
| 라벨 라우팅 / 라벨간 중복 | **RULE**: 카탈로그 labels + gmail_labels 가드 | 카탈로그가 정규 라벨 명시 |
| 중복 제거 (티저↔원문, 동일 link) | **RULE**: link 정규화 dedup | URL 동일성은 코드가 100% 판정 |
| 정크 필터 (바이라인/헤더/목차) | **RULE**(보수적) + **LLM**(프롬프트) | 패턴은 코드, 경계판단은 LLM |
| 렌더모드 (card/list/briefing) | **RULE/CONFIG**: 뉴스레터 유형 메타 | 유형은 발신자별 고정 |
| 인덱스 라우팅 | **RULE** | 정적 경로 |
| 요약 합성/완결성 | **LLM** | 의미 이해 필요 |
| 번역 (영문→한국어) | **LLM**(규칙 강화) | 의역 필요 |
| 같은 사건 의미 병합 (link 다름) | **LLM**(dedup 잔여만) | 의미 동일성 판단 |
| 혼종 CJK 정리 | **RULE**(후처리 감지) + **LLM** | |

## 4. 개발 순서 & 검증

1. **Phase 0 → 1 → 1.5 → 2 → 3 → 4** 순. 각 Phase 후 단위테스트(`npm test`) 통과 확인. (Phase 1.5 누락/오염/소실은 1원칙 위배라 사실상 P0급 — Phase 1과 함께 최우선 처리)
2. **전수 품질 검증**(Phase 1·2 후 필수): CLAUDE.md 프로토콜 —
   - Haiku로 6/1 전 뉴스레터×전 아이템 재추출
   - 원문 메일 1:1 대조: 누락·금지표현·할루시네이션·번역·**출처·중복** 체크
   - 샘플링 금지, 전수.
3. **회귀 방지**: 출처/링크 dedup/정크필터에 단위테스트 추가(`tests/`).
